# 消息队列 (Message Queue)

## 什么是消息队列

消息队列是一种异步通信机制，允许应用将消息发送到队列中，而不需要立即处理，实现系统解耦、异步处理和流量削峰。

```
┌─────────┐     发送消息     ┌─────────┐     消费消息     ┌─────────┐
│ 生产者   │ ──────────────► │  消息队列  │ ──────────────► │  消费者   │
│ Producer│                 │   MQ    │                 │ Consumer│
└─────────┘                 └─────────┘                 └─────────┘
```

## 核心应用场景

### 1. 系统解耦

```
没有 MQ：
订单系统 ──► 库存系统
         ──► 物流系统
         ──► 积分系统
         ──► 短信系统
问题：新增系统需要修改订单系统代码

使用 MQ：
订单系统 ──► 消息队列 ──► 库存系统
                     ──► 物流系统
                     ──► 积分系统
                     ──► 短信系统
优点：新增消费者不影响生产者
```

### 2. 异步处理

```
同步处理：
订单创建 ──► 扣库存(50ms) ──► 发短信(100ms) ──► 加积分(50ms)
总耗时：200ms

异步处理：
订单创建 ──► 发送消息(10ms) ──► 返回成功
                     │
                     ├──► 异步扣库存
                     ├──► 异步发短信
                     └──► 异步加积分
总耗时：10ms
```

### 3. 流量削峰

```
秒杀场景：
瞬间 10万 QPS ──► 消息队列 ──► 系统按 1000 QPS 处理
                （缓冲）      （保护后端系统）
```

## 消息模式

### 点对点 (Point-to-Point)

```
生产者 ──► 队列 ──► 消费者1
              └────► 消费者2（竞争消费）
              └────► 消费者3

特点：
- 每条消息只被一个消费者处理
- 消费者争抢消息
- 适合任务分发
```

### 发布订阅 (Pub/Sub)

```
生产者 ──► Topic ──┬──► 消费者1（订阅者）
              ├──► 消费者2（订阅者）
              └──► 消费者3（订阅者）

特点：
- 每条消息被所有订阅者接收
- 适合广播、事件通知
```

## Kafka 详解

### 核心概念

```
Producer ──► Topic ──► Partition 0 ──► Consumer Group
                │      Partition 1
                │      Partition 2
                │      Partition 3

Topic：消息主题，逻辑概念
Partition：分区，物理存储单位
Offset：消息在分区中的位置
Consumer Group：消费者组，组内消费者共同消费
```

### 分区与副本

```
Topic: order_topic
Partitions: 3
Replication Factor: 2

Broker 1:    [P0 Replica]    [P1 Leader]
Broker 2:    [P0 Leader]     [P2 Replica]
Broker 3:    [P1 Replica]    [P2 Leader]

Leader：负责读写
Replica：同步备份
ISR (In-Sync Replicas)：同步中的副本集合
```

### 生产者策略

```
分区策略：
1. 轮询：round-robin
2. 按 key：hash(key) % partition_count
3. 自定义分区器

ACK 机制：
- acks=0：不等待确认，最高吞吐
- acks=1：等待 Leader 确认
- acks=all：等待所有 ISR 确认，最高可靠性
```

### 消费者组

```
Consumer Group: order_consumers

情况1：4 个消费者，3 个分区
       C1 ──► P0
       C2 ──► P1
       C3 ──► P2
       C4（空闲）

情况2：2 个消费者，3 个分区
       C1 ──► P0, P1
       C2 ──► P2

分区再均衡（Rebalance）：
- 消费者加入/退出时触发
- 重新分配分区所有权
```

## 消息可靠性

### 生产者不丢消息

```
1. 设置 acks=all
2. 设置重试 retries > 0
3. 处理发送失败异常
4. 使用事务（幂等性）
```

### 消息队列不丢消息

```
1. 消息持久化到磁盘
2. 多副本机制（Replication）
3. 同步刷盘（flush）
```

### 消费者不丢消息

```
1. 手动提交 offset（处理成功后提交）
2. 消费幂等设计
3. 死信队列（DLQ）处理失败消息
```

## 消息顺序性

### 全局有序

```
一个 Topic 只有一个 Partition
代价：牺牲吞吐
```

### 分区有序

```
相同 key 的消息发送到同一分区

Producer:
  send("order-1", "create")
  send("order-1", "pay")
  send("order-2", "create")

结果：
  order-1 的消息在同一分区，保证顺序
  order-1 和 order-2 之间不保证顺序
```

## 延迟消息

### 实现方案

```
1. 延迟队列插件（RabbitMQ）
   x-delayed-message 插件

2. 时间轮算法（RocketMQ）
   - 18 个级别（1s, 5s, 10s...2h）
   - 定时扫描投递

3. 独立调度服务
   - 消息存入 DB/Redis
   - 定时任务扫描到期的消息
   - 发送到 MQ 处理
```

## 消息积压处理

```
现象：消费速度 < 生产速度，队列消息堆积

排查：
1. 监控消费速率、生产速率
2. 查看消费者日志
3. 检查是否有消费阻塞

解决：
1. 扩容消费者（增加分区）
2. 优化消费逻辑（批量处理、异步）
3. 跳过非关键消息
4. 临时将消息转移到其他队列
```
