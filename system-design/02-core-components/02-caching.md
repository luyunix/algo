# 缓存 (Caching)

## 什么是缓存

缓存是一种存储机制，用于临时存放可能被频繁访问的数据，以减少对慢速存储（如数据库）的访问，从而提高系统性能。

```
┌─────────────┐         ┌─────────┐         ┌─────────┐
│   客户端     │ ──────► │  缓存   │ ──命中──┤  返回   │
└─────────────┘         └────┬────┘         └─────────┘
                             │ 未命中
                             ▼
                        ┌─────────┐
                        │  数据库  │
                        └────┬────┘
                             │
                             ▼
                        ┌─────────┐
                        │  写入缓存 │
                        └─────────┘
```

## 缓存的层次

```
┌─────────────────────────────────────────┐
│  L1: 浏览器缓存（LocalStorage, Cache API）│
├─────────────────────────────────────────┤
│  L2: CDN 缓存（边缘节点缓存静态资源）      │
├─────────────────────────────────────────┤
│  L3: 反向代理缓存（Nginx/Varnish）        │
├─────────────────────────────────────────┤
│  L4: 应用本地缓存（Caffeine/Guava Cache） │
├─────────────────────────────────────────┤
│  L5: 分布式缓存（Redis/Memcached）        │
├─────────────────────────────────────────┤
│  L6: 数据库缓存（Query Cache, Buffer Pool）│
└─────────────────────────────────────────┘
```

## 缓存更新策略

### 1. Cache Aside（旁路缓存）最常用

```
读取：
1. 先读缓存
2. 缓存命中 → 返回数据
3. 缓存未命中 → 读数据库 → 写入缓存 → 返回数据

写入：
1. 更新数据库
2. 删除缓存（不是更新缓存！）

为什么删除而不是更新？
- 避免并发更新导致的数据不一致
- 删除后下次读取会加载最新数据
```

### 2. Read Through

```
读取：
1. 应用只读缓存
2. 缓存未命中时，缓存组件自动从数据库加载

特点：
- 应用代码简化
- 需要缓存组件支持
```

### 3. Write Through

```
写入：
1. 应用写缓存
2. 缓存同步更新数据库
3. 两者都成功才返回

特点：
- 数据一致性好
- 写延迟增加（等两次写）
```

### 4. Write Behind（Write Back）

```
写入：
1. 应用只写缓存，立即返回
2. 缓存异步批量写入数据库

特点：
- 写性能极高
- 数据可能丢失（缓存宕机）
- 适合写密集型、可容忍短暂不一致的场景
```

## 缓存驱逐策略

当缓存满时，决定哪些数据被淘汰。

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| LRU | 最近最少使用 | 热点数据集中 |
| LFU | 最少频率使用 | 访问频率差异大 |
| FIFO | 先进先出 | 简单场景 |
| TTL | 按过期时间淘汰 | 有时效性数据 |
| Random | 随机淘汰 | 测试场景 |

**LRU 实现**：
```
使用 HashMap + 双向链表
- 访问时移动到队尾
- 淘汰时移除队首
- 时间复杂度：O(1)
```

## 缓存常见问题

### 1. 缓存穿透 (Cache Penetration)

**现象**：查询不存在的数据，每次都要访问数据库。

**原因**：恶意攻击、业务误删。

**解决方案**：
```
1. 布隆过滤器
   - 查询前先用布隆过滤器判断 key 是否存在
   - 不存在直接返回，不查缓存和数据库

2. 缓存空值
   - 数据库不存在也缓存（value=null, TTL=短）
   - 缺点：占用内存，短时间内数据不一致
```

**布隆过滤器原理**：
```
初始化：bit 数组全为 0
添加：用多个 hash 函数计算位置，设为 1
查询：所有位置都是 1 → 可能存在；任一位置为 0 → 一定不存在

特点：
- 可能误判（存在实际不存在）
- 不会漏判（不存在一定报不存在）
- 不能删除
```

### 2. 缓存击穿 (Cache Breakdown)

**现象**：热点 key 过期瞬间，大量请求直达数据库。

**解决方案**：
```
1. 互斥锁
   - 缓存未命中时，加锁查询数据库
   - 只有一个线程能查数据库，其他等待

2. 逻辑过期
   - 缓存永不过期，value 中包含逻辑过期时间
   - 发现过期时异步重建，老数据继续服务

3. 热点 key 永不过期
   - 后台异步更新
```

### 3. 缓存雪崩 (Cache Avalanche)

**现象**：大量 key 同时过期，数据库压力骤增。

**原因**：
- 缓存服务器宕机
- 批量设置的 key 同时过期

**解决方案**：
```
1. 过期时间加随机值
   - 避免同时过期
   - expire_time = base + random(0, 300)

2. 多级缓存
   - L1: 本地缓存（Caffeine）
   - L2: 分布式缓存（Redis）
   - L3: 数据库

3. 熔断降级
   - 数据库压力大时，直接返回默认值或错误提示

4. 缓存高可用
   - Redis Cluster、Sentinel 保证缓存层可用
```

## 缓存一致性

### 问题场景

```
时刻1: 线程A 更新数据库 X=1
时刻2: 线程B 更新数据库 X=2
时刻3: 线程B 删除缓存
时刻4: 线程A 删除缓存（覆盖B的删除）
时刻5: 读请求 → 缓存未命中 → 读数据库 X=1 → 写入缓存

结果：缓存 X=1，数据库 X=2（不一致）
```

### 解决方案

**1. 延迟双删**
```
写操作：
1. 删除缓存
2. 更新数据库
3. 等待 500ms（主从同步时间）
4. 再次删除缓存
```

**2. 监听 Binlog**
```
Canal 监听 MySQL binlog ──► 解析变更 ──► 删除/更新缓存

优点：
- 业务代码无侵入
- 最终一致性
```

**3. 分布式锁**
```
写操作时加分布式锁，保证同一时刻只有一个写操作。
性能较差，适用于强一致性场景。
```

## Redis 缓存实战

### 数据类型选择

| 场景 | 数据类型 | 示例 |
|------|----------|------|
| 对象缓存 | String (JSON) | user:1001 → {...} |
| 用户信息 | Hash | user:1001 field name value "张三" |
| 排行榜 | ZSet | leaderboard:week member user1 score 100 |
| 共同关注 | Set | following:user1 → {user2, user3} |
| 时间线 | List | timeline:user1 → [tweet1, tweet2] |

### 缓存粒度

```
粗粒度：缓存整个对象
- user:1001 → 完整的用户信息 JSON
- 优点：一次获取所有数据
- 缺点：数据量大，部分字段更新需要整个更新

细粒度：缓存字段级别
- user:1001:name → "张三"
- user:1001:age → 25
- 优点：更新灵活，节省内存
- 缺点：需要多次读取

推荐：根据访问模式决定，常一起访问的字段放一起
```

### 大 Key 问题

**危害**：
- 删除时阻塞 Redis（单线程）
- 序列化/反序列化耗时
- 网络传输慢

**解决**：
```
1. 拆分：将大对象拆分成多个小对象
2. 压缩：使用 MessagePack 等序列化
3. 设置合理的过期时间
4. 定期扫描发现并清理：redis-cli --bigkeys
```
