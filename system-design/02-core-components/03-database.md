# 数据库 (Database)

## SQL vs NoSQL 选型

```
┌─────────────────┬─────────────────┐
│      SQL        │     NoSQL       │
├─────────────────┼─────────────────┤
│ 关系型数据       │ 非结构化数据     │
│ 强一致性需求     │ 最终一致性可接受 │
│ 复杂查询/事务    │ 简单查询/高写入  │
│ 垂直扩展为主     │ 水平扩展为主     │
│ Schema 严格      │ Schema 灵活      │
│ MySQL/PostgreSQL│ MongoDB/DynamoDB│
└─────────────────┴─────────────────┘
```

## 数据库优化策略

### 1. 索引优化

**B+ Tree 索引（InnoDB）**：
```
特点：
- 叶子节点存储完整数据
- 非叶子节点存储索引键值
- 适合范围查询、等值查询
- 聚簇索引：数据按主键顺序存储

最左前缀原则：
联合索引 (a,b,c) 可以支持：
- a
- a,b
- a,b,c
不能支持：
- b
- b,c
- c
```

**索引设计原则**：
1. 选择性高的列放前面
2. 经常一起查询的列建联合索引
3. 避免索引过多（影响写入性能）
4. 长字符串用前缀索引

### 2. 查询优化

**EXPLAIN 分析**：
```sql
EXPLAIN SELECT * FROM users WHERE age > 20;

关注字段：
- type: ALL（全表扫描）、index、range、ref、eq_ref、const
- key: 实际使用的索引
- rows: 扫描行数
- Extra: Using filesort（需要优化）、Using index（覆盖索引）
```

**慢查询优化**：
```
1. 开启慢查询日志：long_query_time = 1
2. 定期分析：pt-query-digest
3. 优化方向：
   - 添加合适的索引
   - 优化 SQL 写法（避免 SELECT *）
   - 减少 JOIN 次数
   - 分批处理大数据量
```

## 分库分表

### 何时需要分库分表

| 指标 | 建议值 |
|------|--------|
| 单表数据量 | 超过 500万-1000万 |
| 单表容量 | 超过 10GB |
| 单库 QPS | 超过 5000 |
| 单库连接数 | 接近上限（默认 151）|

### 垂直拆分

**垂直分库**：按业务拆分
```
user_db: 用户相关表
order_db: 订单相关表
product_db: 商品相关表
```

**垂直分表**：按字段拆分
```
用户表：
- 用户基础表：id, name, avatar（常用）
- 用户详情表：id, address, bio（不常用）
```

### 水平拆分

**分片策略**：

1. **哈希取模**
```
slot = hash(user_id) % 8
table = user_0 ~ user_7

优点：数据分布均匀
缺点：扩容时需要迁移数据（rehash）
```

2. **范围分片**
```
user_0: id 1-1000万
user_1: id 1000万-2000万
user_2: id 2000万-3000万

优点：扩容简单
缺点：可能存在热点（新数据集中在最后一张表）
```

3. **一致性哈希**
```
优点：扩容时只影响部分数据
适用：缓存场景
```

### 分库分表的问题

| 问题 | 解决方案 |
|------|----------|
| 分布式事务 | Seata、TCC、最终一致性 |
| 跨分片 JOIN | 全局表、数据冗余、应用层组装 |
| 跨分片排序 | 归并排序、限制返回条数 |
| 全局 ID | 雪花算法、号段模式 |
| 分页查询 | 限制深度、游标分页 |

## 读写分离

### 架构

```
         ┌─────────────┐
         │   写请求     │
         └──────┬──────┘
                ▼
         ┌─────────────┐
         │   主库(Master)│
         │   (写 + 读)   │
         └──────┬──────┘
                │ 复制(Binlog)
       ┌────────┴────────┐
       ▼                 ▼
┌─────────────┐   ┌─────────────┐
│  从库(Slave)  │   │  从库(Slave)  │
│    (只读)     │   │    (只读)     │
└──────┬──────┘   └──────┬──────┘
       └────────┬────────┘
                ▼
         ┌─────────────┐
         │   读请求     │
         └─────────────┘
```

### 复制模式

| 模式 | 描述 | 延迟 |
|------|------|------|
| 异步复制 | Master 不等待 Slave 确认 | 低 |
| 半同步复制 | Master 等待至少一个 Slave 确认 | 中 |
| 同步复制 | Master 等待所有 Slave 确认 | 高 |

### 读写分离的路由

```
1. 基于事务：
   - 有事务 → 走主库
   - 无事务 → 走从库

2. 基于注解/Hint：
   @Master 强制走主库
   @Slave 强制走从库

3. 强制主库场景：
   - 写后立即读（读己之写）
   - 对一致性要求高的查询
```

## 高可用方案

### MySQL 主从切换

```
┌─────────┐      ┌─────────┐
│  Master │ ───► │  Slave  │
│  (故障) │      │ (提升)  │
└─────────┘      └─────────┘

自动切换工具：
- MHA (Master High Availability)
- Orchestrator
- MySQL Router
```

### MySQL 集群

**MGR (MySQL Group Replication)**：
```
基于 Paxos 协议的同步复制
- 多主模式：多个节点可同时写入
- 单主模式：只有一个主节点写入
- 自动故障检测和切换
```

**InnoDB Cluster**：
```
MySQL Shell + MGR + MySQL Router
官方推荐的高可用方案
```

## NoSQL 数据库

### MongoDB

**适用场景**：
- 文档型数据（JSON-like）
- 快速迭代（Schema-less）
- 地理位置查询

**分片机制**：
```
Router (mongos)
     │
     ├─► Config Server（元数据）
     │
     ├─► Shard1（Replica Set）
     ├─► Shard2（Replica Set）
     └─► Shard3（Replica Set）
```

### Redis

**持久化**：
```
RDB：快照，定期保存
- 优点：恢复速度快，文件紧凑
- 缺点：可能丢失最后一次快照后的数据

AOF：日志，记录每个写操作
- 优点：数据更安全（可配置每秒/每次写入）
- 缺点：文件大，恢复慢

混合模式（Redis 4.0+）：
- RDB 做全量，AOF 做增量
```

**高可用**：
```
主从复制 + Sentinel 哨兵

Sentinel 功能：
1. 监控主从节点健康
2. 自动故障转移
3. 通知客户端新主节点地址

Redis Cluster：
- 数据分片（16384 个 slot）
- 无中心架构
- 自动故障转移
```
