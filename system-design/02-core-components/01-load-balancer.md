# 负载均衡 (Load Balancer)

## 什么是负载均衡

负载均衡器是分布式系统的"交通警察"，将传入的流量分发到多个后端服务器，以提高系统的可用性、性能和可扩展性。

```
                    ┌─────────────┐
                    │   客户端     │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  负载均衡器   │
                    │   (LB)      │
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
      ┌─────────┐    ┌─────────┐    ┌─────────┐
      │ Server1 │    │ Server2 │    │ Server3 │
      └─────────┘    └─────────┘    └─────────┘
```

## 负载均衡的层次

### L4 负载均衡（传输层）

- **工作层次**：TCP/UDP 层
- **判断依据**：IP 地址 + 端口号
- **特点**：
  - 性能高（不解析应用层内容）
  - 无法基于 URL/Header 做路由
  - 长连接保持（WebSocket 友好）

```
客户端 ──► LB(192.168.1.1:80) ──► 后端服务器
         只看 IP+Port，不看 HTTP 内容
```

### L7 负载均衡（应用层）

- **工作层次**：HTTP/HTTPS 层
- **判断依据**：URL、Header、Cookie 等
- **特点**：
  - 功能丰富（基于内容路由）
  - 可以做 SSL 终止、压缩、缓存
  - 性能略低于 L4

```
客户端 ──► LB ──► 路由决策
                /api/* ──► API 服务
                /static/* ──► 静态资源服务
                /admin/* ──► 管理服务
```

## 负载均衡算法

### 1. 轮询 (Round Robin)

```
请求1 ──► Server1
请求2 ──► Server2
请求3 ──► Server3
请求4 ──► Server1（循环）
```

**优点**：简单、公平
**缺点**：不考虑服务器负载差异
**适用**：服务器性能相同的场景

### 2. 加权轮询 (Weighted Round Robin)

```
Server1: weight=5
Server2: weight=3
Server3: weight=2

分发序列：1,1,1,1,1, 2,2,2, 3,3
```

**适用**：服务器性能不同的场景

### 3. 最少连接 (Least Connections)

```
新请求发给当前连接数最少的服务器

Server1: 10 connections ──┐
Server2: 5 connections  ──┼──► 发给 Server2
Server3: 8 connections ──┘
```

**适用**：长连接场景（WebSocket、游戏服务器）

### 4. IP 哈希 (IP Hash)

```
server = hash(client_ip) % server_count
```

**优点**：同一 IP 总是路由到同一服务器（会话保持）
**缺点**：某服务器故障时，该 IP 的所有请求都受影响

### 5. 一致性哈希 (Consistent Hashing)

**解决的问题**：
- 普通哈希：server = hash(key) % N
- 当 N 变化时，几乎所有映射都要改变

**原理**：
```
将节点和数据都映射到一个环上（0 ~ 2^32-1）

       hash(node1)
            │
    hash  ──┼──  hash
    (key2)  │   (node2)
            │
            ▼
    ┌───────────────┐
    │               │
    │    哈希环     │
    │   (0-2^32)   │
    │               │
    └───────────────┘
            ▲
            │
       hash(key1)

data 顺时针找到的第一个 node 负责存储
```

**优点**：
- 增加/删除节点只影响相邻数据（约 1/N）
- 适合缓存场景

**虚拟节点**：
- 问题：物理节点少时，数据分布不均匀
- 解决：每个物理节点对应多个虚拟节点

```
真实节点 A ──► 虚拟节点 A1, A2, A3
真实节点 B ──► 虚拟节点 B1, B2, B3
```

## 健康检查

### 目的
及时剔除故障节点，避免将请求发给不可用的服务器。

### 检查方式

| 方式 | 描述 | 粒度 |
|------|------|------|
| Ping | ICMP 探测 | 机器存活 |
| TCP Check | 检测端口是否可连接 | 进程存活 |
| HTTP Check | 访问健康检查接口 | 业务健康 |
| 自定义 | 执行特定脚本 | 详细检查 |

### 检查频率与判定
```
检查间隔：5秒
超时时间：3秒
健康阈值：连续2次成功认为健康
不健康阈值：连续3次失败认为不健康
```

## 高可用方案

### 单点问题
负载均衡器本身成为单点故障。

### 解决方案：主备模式
```
         ┌─────────────┐
         │   Virtual   │
         │    IP       │
         │  (VIP)      │
         └──────┬──────┘
                │
        ┌───────┴───────┐
        ▼               ▼
   ┌─────────┐     ┌─────────┐
   │  LB-Master│     │  LB-Backup│
   │  (Active) │     │  (Standby)│
   └─────────┘     └─────────┘
        │               │
        └───────┬───────┘
                ▼
           后端服务器

使用 Keepalived/VRRP 实现主备自动切换
```

## 面试常见问题

### Q1: 什么是一致性哈希？解决了什么问题？
**答**：一致性哈希将节点和数据映射到一个环上，数据顺时针找到的第一个节点存储。解决了传统哈希在节点增减时大量数据迁移的问题，只影响相邻节点（约 1/N 的数据）。适合缓存场景的负载均衡。

### Q2: L4 和 L7 负载均衡的区别？
**答**：L4 工作在传输层，基于 IP+Port 分发，性能高但不解析应用层内容；L7 工作在应用层，可基于 URL、Header 等做路由决策，功能更丰富但性能略低。

### Q3: 如何实现负载均衡器的高可用？
**答**：使用主备架构配合虚拟 IP（VIP），通过 Keepalived 等工具实现故障自动切换。健康检查剔除故障后端节点，确保流量只发给健康服务器。
