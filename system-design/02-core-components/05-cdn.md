# CDN (内容分发网络)

## 什么是 CDN

CDN（Content Delivery Network）是通过在全球各地部署边缘节点，将网站内容缓存到离用户最近的位置，从而加速内容传输、降低源站压力的网络架构。

```
用户(北京) ──► CDN 节点(北京) ──► 命中缓存 ──► 直接返回 (10ms)
                            │
                            └─► 未命中 ──► 源站(上海) (100ms)

用户(纽约) ──► CDN 节点(纽约) ──► 命中缓存 ──► 直接返回 (20ms)
                            │
                            └─► 未命中 ──► 源站(上海) (300ms)
```

## CDN 的优势

| 优势 | 说明 |
|------|------|
| 加速访问 | 就近访问，降低延迟 |
| 降低带宽 | 减少回源流量，节省成本 |
| 负载均衡 | 分散源站压力 |
| 高可用 | 多节点冗余，故障自动切换 |
| 安全防护 | DDoS 防护、WAF |

## CDN 架构

```
                       ┌─────────────┐
                       │    源站     │
                       │ (Origin)    │
                       └──────┬──────┘
                              │
                    ┌─────────┴─────────┐
                    │    CDN 中心节点    │
                    │  (内容注入/管理)   │
                    └─────────┬─────────┘
                              │
          ┌───────────────────┼───────────────────┐
          ▼                   ▼                   ▼
   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
   │ 边缘节点-华北 │    │ 边缘节点-华东 │    │ 边缘节点-华南 │
   │ (北京/天津)  │    │ (上海/杭州)  │    │ (广州/深圳)  │
   └─────────────┘    └─────────────┘    └─────────────┘
```

## 回源策略

### 1. 直接回源

```
CDN 节点未命中时，直接向源站请求内容。

适用：静态资源、不经常变化的内容
```

### 2. 中间源回源

```
边缘节点 ──► 中间源节点 ──► 源站

优点：
- 减少源站压力
- 边缘节点共享缓存
- 分层缓存策略
```

## 缓存策略

### 缓存键 (Cache Key)

```
默认：URL 完整路径

自定义示例：
- 忽略查询参数：/image.jpg 和 /image.jpg?v=2 视为同一资源
- 包含特定参数：根据语言参数缓存不同版本
- 设备适配：根据 User-Agent 区分移动端/PC端
```

### 缓存控制头

```http
HTTP/1.1 200 OK
Cache-Control: max-age=3600, public
Expires: Wed, 21 Oct 2025 07:28:00 GMT
ETag: "x234dff"
Last-Modified: Wed, 21 Oct 2024 07:28:00 GMT
Vary: Accept-Encoding
```

| 指令 | 说明 |
|------|------|
| max-age | 缓存有效期（秒） |
| no-cache | 可以缓存，但必须重新验证 |
| no-store | 完全禁止缓存 |
| public | 可以被任何缓存存储 |
| private | 只能被浏览器缓存 |
| s-maxage | 共享缓存（CDN）的有效期 |

## CDN 刷新与预热

### 刷新 (Purge)

```
场景：源站内容更新，需要立即让 CDN 缓存失效

方式：
1. URL 刷新：指定具体 URL
2. 目录刷新：刷新整个目录
3. 全站刷新：谨慎使用

注意：
- 刷新有延迟（通常 5-10 分钟）
- 频繁刷新影响 CDN 命中率
```

### 预热 (Preload)

```
场景：大促前提前将热点内容推送到 CDN 节点

方式：
- 提交 URL 列表到 CDN
- CDN 主动从源站拉取内容

优点：
- 避免大促时集中回源压垮源站
- 提升首次访问速度
```

## 动态加速 (DCDN)

### 问题

传统 CDN 只能缓存静态内容，动态请求仍需回源，跨地域延迟高。

### 解决方案

```
用户 ──► 边缘节点 ──► 最优路由 ──► 源站
            │
            └─► 智能路由、TCP 优化、压缩传输

技术：
- Anycast：最近的边缘节点接收请求
- TCP 优化：拥塞控制、连接复用
- 压缩：Gzip/Brotli 压缩传输
- 边缘计算：在边缘节点执行轻量逻辑
```

## CDN 应用场景

| 场景 | 优化手段 |
|------|----------|
| 静态网站 | 全站 CDN 加速 |
| 视频点播 | 分段缓存、自适应码率 |
| 直播 | 推拉流优化、低延迟协议 |
| 下载分发 | 大文件分段、P2P 加速 |
| 电商大促 | 预热、防刷、限流 |

## 面试常见问题

### Q1: 为什么用了 CDN，用户还是访问慢？

**可能原因**：
1. 缓存命中率低（缓存时间短、URL 变化频繁）
2. 未配置 gzip/brotli 压缩
3. 首字节时间（TTFB）高（动态内容未优化）
4. HTTPS 握手慢（未开启 OCSP Stapling）
5. DNS 解析慢（未使用 CDN 的 DNS）

### Q2: 如何防止 CDN 被恶意刷流量？

**方案**：
1. 配置 Referer 防盗链
2. 配置 IP 黑名单/频率限制
3. 使用 URL 鉴权（Token 验证）
4. 监控异常流量告警
