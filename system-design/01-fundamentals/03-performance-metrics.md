# 性能指标

## 延迟 (Latency) vs 吞吐 (Throughput)

```
延迟：完成一个请求所需的时间（毫秒）
吞吐：单位时间内处理的请求数（QPS/TPS）

关系公式：吞吐量 = 并发数 / 平均延迟
```

### 类比理解
- **延迟**：一次旅行的耗时
- **吞吐**：一条公路每小时能通过的车辆数
- **并发**：公路的车道数

### 优化方向
| 问题 | 优化目标 | 常见手段 |
|------|----------|----------|
| 单个请求慢 | 降低延迟 | 缓存、异步、优化算法 |
| 系统承载低 | 提高吞吐 | 水平扩展、批处理、连接池 |

## 常用数字（必须记住）

这些数字是系统设计的"常识"，面试中经常用到：

| 操作 | 时间 | 类比 |
|------|------|------|
| L1 缓存读取 | 1 ns | 一次心跳 |
| L2 缓存读取 | 4 ns | - |
| 内存读取 | 100 ns | 读一页书 |
| SSD 随机读取 | 10 μs | 喝一杯咖啡 |
| 机械磁盘寻道 | 10 ms | 做一次深呼吸 |
| 网络（同机房）| 0.5 ms | - |
| 网络（同城跨机房）| 1-2 ms | - |
| 网络（跨洲）| 150 ms | 眨一次眼 |

**关键结论：**
- 内存比磁盘快 10,000 倍
- 磁盘比网络快（同城）
- 跨洲网络延迟是本地 SSD 的 15,000 倍

## 可用性 (Availability)

### 计算公式
```
可用性 = 正常运行时间 / 总时间
```

### 可用性等级

| 级别 | 可用性 | 年停机时间 | 日停机时间 |
|------|--------|-----------|-----------|
| 1个9 | 90% | 36.5 天 | 2.4 小时 |
| 2个9 | 99% | 3.65 天 | 14.4 分钟 |
| 3个9 | 99.9% | 8.76 小时 | 1.44 分钟 |
| 4个9 | 99.99% | 52.6 分钟 | 8.6 秒 |
| 5个9 | 99.999% | 5.26 分钟 | 0.86 秒 |

### 高可用策略

```
单点故障 ──► 主从复制 ──► 主备切换 ──► 多活架构
 (无)        (读高可用)   (自动故障转移)  (地域级容灾)
```

**常用高可用方案：**
1. **主从复制**：一主多从，主故障时手动/自动切换
2. **多主复制**：多个主节点同时写入
3. **集群模式**：分片 + 副本，自动故障转移
4. **异地多活**：跨地域部署，最高可用性

## 容量估算方法

### 估算步骤

1. **确定流量来源**
   - 日活用户 (DAU)
   - 月活用户 (MAU)
   - 峰值 vs 平均值

2. **计算读写量**
   ```
   读 QPS = DAU × 人均日读次数 / 86400
   写 QPS = DAU × 人均日写次数 / 86400
   峰值 QPS = 平均 QPS × 3-5 倍（根据业务特性）
   ```

3. **计算存储需求**
   ```
   日新增存储 = 日写入量 × 单条数据大小
   总存储 = 日新增 × 保留天数
   ```

4. **计算带宽**
   ```
   入站带宽 = 写 QPS × 平均请求大小
   出站带宽 = 读 QPS × 平均响应大小
   ```

### 估算示例：设计一个微博系统

**假设：**
- MAU: 3 亿
- DAU: 1 亿（MAU 的 1/3）
- 每人每天发 1 条微博
- 每人每天读 100 条微博
- 每条微博平均 500 字节

**计算：**
```
写 QPS：
1亿 × 1 / 86400 ≈ 1157 QPS
峰值 × 3 = 3471 QPS

读 QPS：
1亿 × 100 / 86400 ≈ 115740 QPS
峰值 × 3 = 347222 QPS

存储：
日新增 = 1亿 × 500B = 50GB
5年存储 = 50GB × 365 × 5 ≈ 91TB
（不考虑图片视频）
```

## 性能测试指标

| 指标 | 定义 | 关注点 |
|------|------|--------|
| P50 | 50% 请求低于该值 | 平均体验 |
| P90 | 90% 请求低于该值 | 多数用户体验 |
| P99 | 99% 请求低于该值 | 长尾延迟 |
| P999 | 99.9% 请求低于该值 | 最差情况 |

**为什么关注 P99？**
- 平均数容易被极端值拉偏
- P99 更能反映大多数用户的实际体验
- 微服务架构中，一个慢节点可能影响整个请求链
