# 什么是系统设计

## 定义

系统设计是定义系统架构、组件、接口和其他特征的过程，目的是构建一个满足功能需求且具有良好非功能性特征（性能、可用性、可扩展性）的系统。

## 系统设计的层次

```
┌─────────────────────────────────────┐
│           应用层 (Application)        │  ← 业务逻辑、用户界面
├─────────────────────────────────────┤
│           服务层 (Services)           │  ← API 网关、微服务
├─────────────────────────────────────┤
│           数据层 (Data)               │  ← 数据库、缓存、消息队列
├─────────────────────────────────────┤
│           基础设施层 (Infrastructure)  │  ← 服务器、网络、CDN
└─────────────────────────────────────┘
```

## 面试中的 4S 框架

### 1. Scope（明确范围）- 5分钟
**需要澄清的问题：**
- 功能需求：核心功能是什么？有哪些附加功能？
- 非功能需求：预期的 QPS？数据量？延迟要求？
- 用户规模：DAU/MAU 多少？
- 地理分布：单区域还是全球？

**容量估算示例：**
```
假设设计一个短链接系统：
- DAU: 100万
- 每人每天创建 1 个短链
- 每个短链被访问 100 次
- 读写比: 100:1

计算：
- 日新增短链: 100万
- 日跳转请求: 1亿
- 写 QPS: 100万/86400 ≈ 12
- 读 QPS: 1亿/86400 ≈ 1157
- 存储: 100万 × 1KB = 1GB/天
```

### 2. Sketch（高层设计）- 10分钟
**产出物：**
- 系统架构图
- API 设计（RESTful）
- 数据模型（ER 图）
- 基本数据流

### 3. Solidify（详细设计）- 15分钟
**深入点：**
- 核心组件的实现细节
- 数据库选型与分片策略
- 缓存策略
- 一致性保障

### 4. Scale（扩展性分析）- 10分钟
**考虑因素：**
- 当前瓶颈分析
- 水平扩展方案
- 故障处理机制
- 监控与告警

## 常见误区

| 误区 | 正确做法 |
|------|----------|
| 一开始就深入细节 | 先给高层设计，再逐步深入 |
| 忽视非功能需求 | 主动询问 QPS、延迟、可用性要求 |
| 过度设计 | 先满足当前需求，再考虑扩展 |
| 不与面试官沟通 | 持续交流，确认假设 |
