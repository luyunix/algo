# 一致性模型

## 一致性强度谱

```
强一致性 ────────────────────────► 最终一致性
（越强越难实现，性能越低）

强一致性 > 顺序一致性 > 因果一致性 > 会话一致性 > 最终一致性
```

## 强一致性 (Strong Consistency)

### 定义
任何读取操作都能读到最新写入的数据。所有节点看到的数据顺序完全一致。

### 实现方式
- **两阶段提交 (2PC)**：准备 + 提交
- **Paxos/Raft**：共识算法
- **同步复制**：写入所有副本才返回成功

### 代价
- 高延迟：需要协调多个节点
- 低可用：部分节点故障时无法写入

### 适用场景
- 银行转账
- 库存扣减
- 配置中心

## 顺序一致性 (Sequential Consistency)

### 定义
所有进程看到的操作顺序一致，但不需要和全局时钟一致。

```
进程A：W(x,1) ──────► W(x,2)
进程B：──────── R(x,1) ────► R(x,2)

允许的：所有进程都看到 1 先于 2
不允许：A 看到 1先于2，B 看到 2先于1
```

## 因果一致性 (Causal Consistency)

### 定义
如果操作 A 因果影响操作 B，那么所有节点必须先看到 A 再看到 B。无关的操作可以乱序。

```
因果关系的例子：
用户A评论了帖子 → 用户B回复了这条评论
（有因果关系，必须有序）

用户A点赞了帖子 → 用户B点赞了另一个帖子
（无因果关系，可以乱序）
```

## 会话一致性 (Session Consistency)

### 定义
保证单个用户的操作在自己的会话内是一致的（读己之写）。

```
用户写入 X=1 ──► 同一用户立即读取 X → 得到 1
其他用户可能暂时看到 X=0
```

### 实现方式
- 会话粘滞：同一用户路由到同一服务器
- 版本号验证：读取时检查版本
- 短暂读主库：写后短时间内读主库

## 最终一致性 (Eventual Consistency)

### 定义
不保证立即读到最新数据，但保证在没有新写入的情况下，最终所有副本会一致。

### 特点
- 高可用：任何副本都可以响应读请求
- 高性能：异步复制
- 可能读到旧数据

### 适用场景
- DNS 系统
- 社交网络 Feed
- 电商商品详情

## 读写一致性保障

### Read Your Writes（读己之写）

```
问题：用户发布微博后，刷新看不到自己的微博
原因：写入主库，但从从库读取

解决方案：
1. 写后短时间读主库（如：写后1秒内）
2. 客户端缓存自己写入的数据
3. 使用会话粘滞
```

### Monotonic Reads（单调读）

```
问题：用户先读到评论 v2，刷新后变成 v1（更旧的）
原因：从不同的副本读取

解决方案：
1. 会话粘滞到同一副本
2. 版本号检查，拒绝旧版本
```

### Consistent Prefix Reads（一致前缀读）

```
问题：先看到"回复B"，后看到"评论A"（A是B的父评论）
原因：复制延迟导致乱序

解决方案：
1. 因果跟踪（vector clock）
2. 等待相关数据到达后再返回
```

## 一致性模型选择指南

| 场景 | 推荐模型 | 原因 |
|------|----------|------|
| 银行余额查询 | 强一致性 | 钱不能算错 |
| 微博 Feed 流 | 最终一致性 | 短暂不一致可接受 |
| 微信消息 | 会话一致性 | 自己发的消息要能立即看到 |
| 电商购物车 | 最终一致性 | 可用性优先 |
| 配置下发 | 强一致性 | 配置错误影响全局 |

## 实现强一致性的技术

### 1. 两阶段提交 (2PC)

```
协调者                  参与者
   │    1. Prepare     │
   │ ────────────────> │
   │    2. Yes/No      │
   │ <──────────────── │
   │    3. Commit      │
   │ ────────────────> │
   │    4. Ack         │
   │ <──────────────── │

问题：协调者单点故障、同步阻塞
```

### 2. Raft 共识算法

```
Leader Election ──► Log Replication ──► Safety
（选主）            （日志复制）         （安全保证）

特点：
- 强领导者：所有写入通过 Leader
- 日志顺序一致
- 大多数节点确认即提交

应用：etcd、Consul、TiKV
```

### 3. Quorum NWR 机制

```
N：副本数
W：写入确认数
R：读取确认数

规则：W + R > N 时保证强一致性

示例：N=3, W=2, R=2
- 写入时等待 2 个副本确认
- 读取时读取 2 个副本，取最新版本
- 2 + 2 > 3，保证至少有一个重叠
```
