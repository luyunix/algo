# 可观测性 (Observability)

## 什么是可观测性

可观测性是通过系统的外部输出（日志、指标、追踪）来理解系统内部状态的能力。

```
传统监控：知道系统出问题了
可观测性：知道系统为什么出问题

三大支柱：
┌─────────┐ ┌─────────┐ ┌─────────┐
│  Logs   │ │ Metrics │ │ Traces  │
│  (日志)  │ │ (指标)   │ │ (追踪)   │
└─────────┘ └─────────┘ └─────────┘
```

## 日志 (Logging)

### 日志级别

| 级别 | 使用场景 |
|------|----------|
| DEBUG | 调试信息，生产环境通常关闭 |
| INFO | 正常流程信息 |
| WARN | 警告，可能有问题但继续运行 |
| ERROR | 错误，需要处理但程序继续 |
| FATAL | 致命错误，程序退出 |

### 结构化日志

```
非结构化：
2024-01-01 12:00:00 User login failed: username=admin, reason=wrong_password

结构化（JSON）：
{
  "timestamp": "2024-01-01T12:00:00Z",
  "level": "ERROR",
  "message": "User login failed",
  "user_id": "admin",
  "reason": "wrong_password",
  "ip": "192.168.1.1",
  "trace_id": "abc123"
}
```

### 日志收集架构

```
应用日志 ──► Filebeat ──► Kafka ──► Logstash ──► Elasticsearch ──► Kibana
                                      │
                                      ▼
                                长期存储 (S3)
```

### 最佳实践

```
1. 统一格式：使用结构化日志（JSON）
2. 关联标识：每条日志包含 trace_id
3. 上下文信息：包含用户ID、请求ID等
4. 避免敏感信息：密码、Token 脱敏
5. 合理采样：DEBUG 日志采样，ERROR 全量
```

## 指标 (Metrics)

### 指标类型

| 类型 | 说明 | 示例 |
|------|------|------|
| Counter | 累计值，只增不减 | 请求总数、错误数 |
| Gauge | 瞬时值，可增可减 | CPU 使用率、内存使用 |
| Histogram | 采样分布 | 请求延迟分布 |
| Summary | 分位数统计 | P99 延迟 |

### RED 方法

```
面向请求的监控：
- Rate（速率）：每秒请求数
- Errors（错误）：错误率
- Duration（延迟）：请求处理时间
```

### USE 方法

```
面向资源的监控：
- Utilization（使用率）：资源使用百分比
- Saturation（饱和度）：排队工作量
- Errors（错误）：错误计数
```

### Prometheus + Grafana

```
Prometheus：时序数据库，拉取模式采集指标
Grafana：可视化面板

架构：
┌─────────┐     pull      ┌─────────┐
│   App   │◄──────────────│Prometheus│
│ (exporter)│              └────┬────┘
└─────────┘                     │
                                ▼
                          ┌─────────┐
                          │ Grafana │
                          │(可视化) │
                          └─────────┘
```

## 追踪 (Tracing)

### 分布式追踪

```
跟踪请求在微服务间的完整链路：

请求 ──► API Gateway ──► Order Service ──► Database
              │               │
              │               ▼
              │          Payment Service
              │               │
              ▼               ▼
         Trace ID: abc-123-xyz（贯穿整个请求链）

每个服务生成 Span，记录：
- 开始时间、结束时间
- 耗时
- 标签（服务名、方法名等）
- 日志事件
```

### OpenTelemetry

```
CNCF 项目，统一的观测标准：
- 统一的 API 和 SDK
- 自动埋点（常见框架）
- 支持多种后端（Jaeger、Zipkin、Prometheus）
```

### 采样策略

```
全量收集成本太高，需要采样：

1. 头部采样（Head-based）：
   - 请求开始时决定是否采样
   - 简单但可能漏掉重要请求

2. 尾部采样（Tail-based）：
   - 请求完成后根据特征决定是否保留
   - 可以保留错误请求、慢请求
   - 需要缓存所有 Span，内存压力大
```

## 告警

### 告警分级

| 级别 | 响应时间 | 通知方式 | 场景 |
|------|----------|----------|------|
| P0 | 5分钟 | 电话+短信 | 核心业务故障 |
| P1 | 15分钟 | 短信+钉钉 | 重要功能异常 |
| P2 | 1小时 | 钉钉+邮件 | 一般告警 |
| P3 | 次日 | 邮件 | 提醒类 |

### 告警原则

```
1.  actionable：每个告警都应该有可执行的操作
2.  抑制：故障根因未解决时，抑制衍生告警
3.  分级：不同级别不同响应
4.  收敛：相似告警合并
5.  定期清理：无效告警及时删除
```

## 可观测性最佳实践

### 日志规范

```
必含字段：
- timestamp：ISO8601 格式
- level：日志级别
- service：服务名
- trace_id：追踪ID
- message：日志内容

可选字段：
- user_id：用户ID
- request_id：请求ID
- duration：耗时
- error_type：错误类型
```

### 指标设计

```
命名规范：{service}_{resource}_{metric}_{unit}
示例：
- http_requests_total
- http_request_duration_seconds
- database_connections_active

Label 设计：
- 控制 cardinality（标签组合数）
- 高 cardinality 的标签不要用（如 user_id）
```

### 追踪埋点

```
自动埋点：
- HTTP 请求/响应
- 数据库调用
- 缓存调用
- RPC 调用
- 消息队列

手动埋点：
- 业务关键路径
- 复杂计算
- 外部 API 调用
```

## 面试要点

### Q1: 日志、指标、追踪有什么区别？什么时候用什么？

**答**：
- 日志：记录离散事件，适合排查具体问题，如错误详情
- 指标：聚合统计，适合看趋势、告警，如 QPS、延迟
- 追踪：请求链路，适合分析跨服务调用性能

三者互补，完整的可观测性需要三者结合。

### Q2: 如何处理高并发场景下的日志性能问题？

**答**：
1. 异步日志：使用队列缓冲，不阻塞业务线程
2. 日志分级：生产环境关闭 DEBUG
3. 日志采样：DEBUG 日志按比例采样
4. 批量发送：收集一定量或一定时间后批量发送
5. 本地缓存：网络故障时缓存到本地

### Q3: 如何设计一个有效的告警系统？

**答**：
1. 遵循 RED/USE 方法，覆盖关键指标
2. 分级告警，不同级别不同响应方式
3. 告警要有可操作性，避免无效告警
4. 告警抑制和收敛，避免告警风暴
5. 定期 review，清理无效告警
