# 微服务架构

## 什么是微服务

微服务架构是将单体应用拆分为一组小型服务的方法，每个服务运行在独立的进程中，通过轻量级机制通信。

```
单体架构：
┌─────────────────────────────────────────┐
│           单体应用 (Monolith)            │
│  ┌────────┐┌────────┐┌────────┐        │
│  │ 用户模块 ││ 订单模块 ││ 库存模块 │        │
│  └────────┘└────────┘└────────┘        │
│  ┌──────────────────────────────────┐  │
│  │           共享数据库               │  │
│  └──────────────────────────────────┘  │
└─────────────────────────────────────────┘

微服务架构：
┌───────────┐ ┌───────────┐ ┌───────────┐
│  用户服务  │ │  订单服务  │ │  库存服务  │
│ (独立部署) │ │ (独立部署) │ │ (独立部署) │
└─────┬─────┘ └─────┬─────┘ └─────┬─────┘
      │             │             │
      └─────────────┼─────────────┘
                    │
            ┌───────┴───────┐
            │   API Gateway  │
            └───────┬───────┘
                    │
                ┌───┴───┐
                │  客户端  │
                └───────┘
```

## 微服务 vs 单体

| 特性 | 单体 | 微服务 |
|------|------|--------|
| 部署 | 简单 | 复杂 |
| 扩展 | 整体扩展 | 独立扩展 |
| 技术栈 | 统一 | 多样 |
| 团队 | 耦合 | 独立 |
| 故障隔离 | 差 | 好 |
| 数据一致性 | 简单 | 复杂 |
| 运维成本 | 低 | 高 |

## 服务拆分原则

### 拆分方法

```
1. 按业务领域 (DDD)
   - 用户域、订单域、商品域

2. 按职能
   - 接入层、业务层、数据层

3. 按变更频率
   - 稳定服务 vs 常变服务分开
```

### 拆分粒度

```
过粗：
- 仍是单体思维，获益不明显

过细：
- 服务爆炸，运维困难
- 分布式事务问题增多

推荐：
- 两个披萨原则（团队规模）
- 康威定律：组织架构决定系统架构
```

## 核心组件

### API Gateway

```
功能：
- 统一入口：客户端无需知道后端服务地址
- 路由：/users → 用户服务，/orders → 订单服务
- 认证鉴权：统一身份验证
- 限流熔断：保护后端服务
- 协议转换：HTTP ↔ gRPC

实现：
- Kong
- Spring Cloud Gateway
- Nginx + Lua
```

### 服务注册与发现

```
见 03-distributed-systems/04-service-discovery.md
```

### 配置中心

```
集中管理配置：
- 统一管理各服务配置
- 动态刷新（无需重启）
- 版本管理、灰度发布

实现：
- Nacos
- Apollo
- Consul
- Spring Cloud Config
```

### 服务通信

| 方式 | 协议 | 特点 | 场景 |
|------|------|------|------|
| REST | HTTP/JSON | 简单、通用 | 对外 API |
| gRPC | HTTP/2 + Protobuf | 高性能、强类型 | 内部通信 |
| Dubbo | TCP + Hessian | 丰富特性 | Java 生态 |
| 消息队列 | 自定义 | 异步解耦 | 不需要实时响应 |

### 熔断降级

```
熔断器模式：
CLOSED ──► OPEN ──► HALF_OPEN ──► CLOSED
（正常）   （熔断）    （半开）      （恢复）

触发条件：
- 错误率超过阈值
- 连续失败次数超过阈值

降级策略：
- 返回默认值
- 返回缓存数据
- 返回简化版服务
```

**实现**：
```java
// Hystrix/Sentinel
@HystrixCommand(
    fallbackMethod = "getUserFallback",
    commandProperties = {
        @HystrixProperty(name = "circuitBreaker.requestVolumeThreshold", value = "10"),
        @HystrixProperty(name = "circuitBreaker.errorThresholdPercentage", value = "50"),
        @HystrixProperty(name = "circuitBreaker.sleepWindowInMilliseconds", value = "10000")
    }
)
public User getUser(Long id) {
    return userServiceClient.getUser(id);
}

public User getUserFallback(Long id) {
    return User.defaultUser();
}
```

## 数据一致性

### 分布式事务

```
见 03-distributed-systems/03-distributed-transaction.md

微服务推荐：
- 能不用就不用
- 优先使用消息队列最终一致
- 核心链路使用 TCC/Saga
```

### 数据分片策略

```
按服务分库：
- 用户服务：user_db
- 订单服务：order_db
- 商品服务：product_db

跨服务查询：
- API 组装：调用多个服务接口
- CQRS：读写分离，读服务直连数据库
- 数据冗余：在需要的地方冗余数据
```

## 服务网格 (Service Mesh)

### 概念

```
将服务通信逻辑从业务代码中剥离，下沉到基础设施层。

Before:
应用代码 = 业务逻辑 + 服务发现 + 负载均衡 + 熔断 + 监控

After:
应用代码 = 业务逻辑
Sidecar = 服务发现 + 负载均衡 + 熔断 + 监控
```

### 架构

```
┌─────────┐      ┌─────────┐
│   App   │◄────►│ Sidecar │
└─────────┘      └────┬────┘
                      │
                      ▼
               ┌─────────────┐
               │  Service Mesh │
               │   (Control Plane) │
               └─────────────┘

Sidecar 实现：Envoy、Linkerd
控制平面：Istio、Consul Connect
```

### Istio 功能

```
- 流量管理：路由、分流、灰度发布
- 安全：mTLS、认证授权
- 可观测性：指标、日志、追踪
```

## 微服务最佳实践

### 设计原则

```
1. 单一职责：一个服务只负责一个业务领域
2. 高内聚低耦合：服务内部紧密，服务间松散
3. 接口契约：明确定义接口，向后兼容
4. 容错设计：假设依赖服务会失败
5. 可观测性：日志、指标、追踪
```

### 部署策略

| 策略 | 说明 | 风险 |
|------|------|------|
| 滚动更新 | 逐个替换实例 | 低 |
| 蓝绿部署 | 两套环境切换 | 中 |
| 金丝雀发布 | 小部分流量测试 | 低 |
| A/B 测试 | 对比两个版本 | 中 |

### 组织架构

```
康威定律：设计系统的组织，其产生的设计等于组织之间的沟通结构。

推荐：
- 跨职能团队（产品 + 开发 + 测试 + 运维）
- 每个团队负责完整的微服务生命周期
- 团队间通过 API 契约协作
```

## 面试要点

### Q1: 什么情况下应该用微服务？

**答**：
- 团队规模较大（>30 人）
- 业务复杂度高
- 需要独立扩展不同模块
- 需要多技术栈
- 团队有足够运维能力

早期小团队建议先用单体，业务复杂后再拆分。

### Q2: 微服务的数据一致性怎么保证？

**答**：
- 优先使用最终一致性（消息队列）
- 强一致性场景使用 TCC/Saga
- 避免分布式事务，通过业务设计减少跨服务事务
- 接受短暂不一致，保证最终一致

### Q3: 微服务如何测试？

**答**：
- 单元测试：服务内部逻辑
- 集成测试：服务间交互（契约测试）
- 端到端测试：完整流程
- 使用 Mock 模拟依赖服务
- 混沌工程：注入故障测试容错
