# 设计短链接系统 (TinyURL)

## 需求分析

### 功能需求

1. **短链接生成**：将长链接转换为短链接
2. **短链接跳转**：访问短链接时重定向到原始链接
3. **自定义短码**（可选）：允许用户自定义短码
4. **链接统计**（可选）：点击量、访问分布等

### 非功能需求

| 需求 | 目标 |
|------|------|
| 可用性 | 99.99% |
| 延迟 | 跳转延迟 < 100ms |
| 短码长度 | 7-8 个字符 |

## 容量估算

### 假设

```
- DAU: 1000 万
- 每人每天创建 1 个短链
- 每个短链每天被访问 100 次
- 读写比: 100:1
- 服务周期: 10 年
```

### 计算

```
日新增短链: 1000 万
年新增: 36.5 亿
10 年总量: 36.5 亿

存储估算：
- 短码: 7 bytes
- 长链接: 平均 500 bytes
- 元数据（创建时间等）: 100 bytes
- 单条记录: ~600 bytes

10 年存储: 36.5 亿 × 600B ≈ 219 GB
考虑索引、备份: ~1 TB

QPS 估算：
- 创建 QPS: 1000万/86400 ≈ 116
- 跳转 QPS: 1000万×100/86400 ≈ 11,574
- 峰值 QPS (×3): 创建 350, 跳转 35,000
```

## 数据库设计

```sql
CREATE TABLE short_url (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    short_code VARCHAR(10) UNIQUE NOT NULL COMMENT '短码',
    long_url VARCHAR(2048) NOT NULL COMMENT '原始链接',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    expires_at TIMESTAMP NULL COMMENT '过期时间',
    click_count INT DEFAULT 0 COMMENT '点击次数',
    INDEX idx_short_code (short_code)
) ENGINE=InnoDB;

-- 防止重复提交的唯一索引
CREATE TABLE url_mapping (
    url_hash CHAR(32) PRIMARY KEY COMMENT '长链接MD5',
    short_code VARCHAR(10) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 短码生成方案

### 方案 1：哈希算法

```
步骤：
1. 长链接做 MD5
2. 取前 8 位
3. Base62 编码 (a-zA-Z0-9)

问题：
- 哈希冲突（不同的长链可能生成相同的短码）
- 冲突解决：冲突时加盐重新哈希
```

### 方案 2：自增 ID + Base62

```
步骤：
1. 数据库自增 ID
2. 将 ID 转为 Base62 (0-9a-zA-Z)

示例：
ID: 100000000
Base62: 26gR

优点：无冲突、趋势递增
缺点：短码连续，可预测
改进：加随机偏移或跳号
```

### 方案 3：雪花算法

```
使用分布式 ID 生成器
优点：全局唯一、高性能
缺点：ID 较长，需要 Base62 压缩
```

### 方案 4：预生成（推荐）

```
1. 预先批量生成短码，存入数据库
2. 使用时直接取出，标记为已使用

优点：
- 生成和使用解耦
- 可以过滤易混淆字符（0O, 1lI）
- 可以预加载到缓存
```

## 核心流程

### 创建短链

```
1. 检查长链接合法性（长度、格式）
2. 布隆过滤器检查是否已存在
   - 可能存在 → 查数据库确认
   - 确定不存在 → 继续
3. 获取短码（从发号器）
4. 写入数据库
5. 写入 Redis 缓存
6. 返回短链
```

### 跳转流程

```
1. 从 URL 解析短码
2. 查 Redis
   - 命中 → 302 重定向
   - 未命中 → 查数据库
3. 数据库命中 → 写入 Redis → 302 重定向
4. 数据库未命中 → 404
```

### 重定向类型

```
301 - 永久重定向：
- 浏览器会缓存结果
- 下次直接访问原链接，不再请求短链服务
- 减轻服务器压力，但无法统计点击

302 - 临时重定向：
- 浏览器不缓存
- 每次都要请求短链服务
- 可以统计点击，服务器压力大

推荐：302，便于统计和后期修改
```

## 优化策略

### 1. 缓存优化

```
Redis 缓存策略：
- 热点数据永不过期
- 冷数据 LRU 淘汰
- 缓存穿透：布隆过滤器 + 缓存空值

多级缓存：
- L1: CDN（如果被浏览器缓存）
- L2: 本地缓存（Caffeine）
- L3: Redis
- L4: 数据库
```

### 2. 读写分离

```
写操作 → 主库
读操作 → 从库

主从复制延迟问题：
- 创建后立即跳转的场景较少
- 可以接受短暂延迟
```

### 3. 数据库分片

```
按短码首字符分片：
- short_code[0] in 'a-h' → DB1
- short_code[0] in 'i-p' → DB2
- short_code[0] in 'q-z' → DB3
- short_code[0] in '0-9A-Z' → DB4

或者按 ID 哈希取模
```

## 安全问题

### 1. 短链伪造

```
风险：遍历短码，访问所有短链

防护：
- 短码长度足够（7-8 位，62^7 ≈ 3.5 万亿）
- 添加权限校验（私有短链需要登录）
- 速率限制
```

### 2. 恶意链接

```
风险：短链指向钓鱼网站、病毒链接

防护：
- 链接安全检测（黑名单、内容扫描）
- 用户举报机制
- 跳转前警告页面
```

### 3. 滥用

```
风险：批量生成短链，占用资源

防护：
- 用户限流（每个用户每天最多创建 N 个）
- IP 限流
- 需要登录才能创建
```

## 面试要点

### Q1: 如何设计一个可以支持 10 亿级短链的系统？

**答**：
1. 使用预生成短码方案，避免生成冲突
2. 多级缓存：本地缓存 + Redis + CDN
3. 数据库分片，按短码或 ID 分片
4. 读写分离，主从复制
5. 布隆过滤器防止缓存穿透
6. 支持 301/302 重定向，平衡性能和统计需求

### Q2: 短码生成如何保证不重复且高效？

**答**：
1. 使用号段模式，批量从数据库获取 ID
2. 本地缓存一批短码，避免每次请求数据库
3. 雪花算法作为备选方案
4. 双 Buffer 机制，保证平滑切换

### Q3: 短链服务如何防止被恶意刷量？

**答**：
1. 限流：IP 限流、用户限流
2. 验证：验证码、登录验证
3. 监控：异常流量检测
4. 熔断：数据库压力大时拒绝服务
