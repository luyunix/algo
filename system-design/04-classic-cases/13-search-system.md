# 搜索系统设计

从工程视角深入搜索系统架构，涵盖倒排索引、Query 理解、相关性排序等核心模块。

## 1. 搜索 vs 推荐的区别

```
┌─────────────────┬─────────────────┐
│     搜索        │     推荐        │
├─────────────────┼─────────────────┤
│ 用户有明确意图   │ 用户意图不明确   │
│ 主动查询        │ 被动接收        │
│ Query 驱动      │ 用户画像驱动    │
│ 相关性优先      │ 个性化优先      │
│ 结果数有限      │ 可以无限滚动    │
│ 容错率低        │ 容错率较高      │
└─────────────────┴─────────────────┘

工程差异：
- 搜索对延迟更敏感（用户等待）
- 搜索需要更复杂的 Query 理解
- 搜索需要支持布尔查询、过滤条件
- 搜索需要处理长尾 Query
```

## 3. 倒排索引 (Inverted Index)

### 3.1 基本结构

```
文档集合：
Doc1: "苹果 手机 新款"
Doc2: "苹果 电脑 笔记本"
Doc3: "华为 手机 新款"

倒排索引：
苹果     → [Doc1, Doc2]
手机     → [Doc1, Doc3]
新款     → [Doc1, Doc3]
电脑     → [Doc2]
笔记本   → [Doc2]
华为     → [Doc3]

进阶（带位置信息）：
苹果     → [(Doc1, pos:0), (Doc2, pos:0)]
手机     → [(Doc1, pos:1), (Doc3, pos:1)]
```

### 3.2 索引构建流程

```
原始数据（MySQL/Binlog）

         ▼
┌─────────────────┐
│   数据预处理     │
│  - 分词         │
│  - 去停用词     │
│  - 词干提取     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   索引构建       │
│  - 内存构建      │
│  - 段合并        │
│  - 索引优化      │
└────────┬────────┘
         │

┌─────────────────┐
│   索引分发       │
│  - 主从同步      │

└─────────────────┘
```

### 3.3 分词器选择

| 分词器 | 适用场景 | 特点 |
|--------|----------|------|
| IK Analyzer | 中文搜索 | 细粒度/智能分词 |
| Jieba | 快速原型 | Python 生态 |
| HanLP | 高精度 | NLP 功能丰富 |
| 自建分词 | 垂直领域 | 自定义词典 |

```java
// 中文分词示例
public class ChineseTokenizer {
    public List<Token> tokenize(String text) {
        // 1. 粗分（基于词典）
        List<Token> tokens = roughSegment(text);

        // 2. 消歧（选择最佳切分路径）
        tokens = disambiguate(tokens);

        // 3. 识别专有名词（NER）
        tokens = recognizeNamedEntity(tokens);

        // 4. 同义词扩展
        tokens = expandSynonyms(tokens);

        return tokens;
    }
}
```

### 3.4 索引压缩

```
文档列表压缩（PForDelta）：
- 原始：[100, 105, 110, 115] → 占用 16 bytes
- 差值：[100, 5, 5, 5] → 占用 7 bytes
- 配合变长编码（VByte/Golomb）进一步压缩

词频压缩：
- Gamma 编码
- Golomb 编码

优点：
- 减少内存占用
- 提高缓存命中率
- 加速倒排列表遍历
```

```

│  Text   │          │  Text   │
```

### 4.2 混合召回架构

```
搜索请求
    │
    ├─► 倒排索引召回 ──┐

    │                  │

融合策略：

3. 机器学习排序（LTR）
```

## 5. Query Understanding

### 5.1 Query 纠错

```
类型：
1. 拼写错误："iphnoe" → "iphone"
2. 拼音错误："pingguo" → "苹果"
3. 简写："kouhong" → "口红"
4. 别字："衣服" → "衣服"

算法：
- 编辑距离（Edit Distance）
- 噪声信道模型（Noisy Channel）
- 基于 BERT 的纠错

实现：
- 前缀树（Trie）快速前缀匹配
- 贝叶斯推断选择最可能的正确词
```

### 5.2 Query 扩展

```
方式：
1. 同义词扩展
   "手机" → "手机", "移动电话", "手提电话"

2. 上下位词扩展
   "水果" → "苹果", "香蕉", "橙子"

3. 相关搜索词
   "iPhone" → "iPhone 15", "iPhone 价格", "iPhone 评测"

4. 个性化扩展（基于用户历史）

实现：
- 基于知识图谱（WordNet/ConceptNet）
- 基于用户行为挖掘（共现统计）
- 基于 Seq2Seq 生成
```

### 5.3 意图识别

```
意图分类：
├── 导航类："淘宝官网", "京东登录"
├── 信息类："天气预报", "股票行情"
├── 交易类："买 iPhone", "订机票"
├── 本地类："附近餐厅", "北京酒店"
└── 问答类："如何做蛋糕", "什么是区块链"

实现方式：
1. 规则匹配（关键词）
2. 文本分类（FastText/BERT）
3. 槽位填充（Slot Filling）

应用：
- 触发不同垂类搜索（商品/新闻/图片）
- 展示不同卡片样式
- 调用不同排序模型
```

## 6. 相关性排序

### 6.1 传统方法：TF-IDF + BM25

```
TF (Term Frequency):
词在文档中出现的频率
TF = 词在文档中出现次数 / 文档总词数

IDF (Inverse Document Frequency):
词的区分度
IDF = log(总文档数 / 包含词的文档数)

TF-IDF = TF × IDF

BM25 (改进版):
score(Q, D) = Σ IDF(q_i) × [f(q_i, D) × (k1 + 1)] / [f(q_i, D) + k1 × (1 - b + b × |D|/avgDL)]

参数：
- k1: 控制词频饱和度（通常 1.2-2.0）
- b: 控制文档长度归一化（通常 0.75）

优点：
- 可解释性强
- 计算快
- 不需要训练数据
```

### 6.2 学习型稀疏检索 (Learned Sparse Retrieval)

```
代表：DeepCT, docT5query

思想：
- 用深度学习预测词的重要性权重
- 替代传统的 TF，成为 "Semantic TF"

流程：
Doc → BERT → 每个词的权重 → 构建加权倒排索引

Query → 分词 → 检索加权索引 → 汇总得分

优点：
- 结合了语义理解和倒排索引的效率
- 可解释性比向量检索好
```

### 6.3 深度排序模型

```
架构：
Query ──┐
        ├──► Feature Cross ──► DNN ──► 相关性分数
Doc  ───┘

特征：
- 文本匹配：BM25、编辑距离、共现词数
- 语义匹配：向量相似度
- 点击特征：CTR、历史点击关系
- 用户特征：用户画像（个性化）

模型演进：
DSSM → CDSSM → ARC-I/ARC-II → MatchPyramid → KNRM → Conv-KNRM
```

## 7. 性能优化

### 7.1 查询优化

```
1. Query 缓存：
   - 缓存热门 Query 的结果
   - 命中率通常 30-50%
   - TTL 根据业务设置（5-30 分钟）

2. 结果缓存：
   - 缓存倒排列表的合并结果
   - 适用于分页查询

3. 预计算：
   - 热门 Query 提前计算
   - 推送至 CDN 或本地缓存
```

### 7.2 倒排索引优化

```
1. 跳表 (Skip List)：
   - 加速倒排列表的求交集
   - 时间复杂度从 O(n+m) 降到 O(√n + √m)

2. 分层索引：
   - 热词索引常驻内存
   - 冷词索引按需加载

3. 索引分片：
   - 按文档 ID 范围分片
   - 并行检索后合并

4. 布隆过滤器：
   - 快速判断词是否存在
   - 减少无效磁盘 IO
```

### 7.3 召回优化

```
1. 多路召回裁剪：
   - 每路召回限制数量（如倒排召回最多 1000）
   - 按质量分数动态调整

2. 动态 pruning：
   - WAND (Weak AND)
   - Block-Max WAND (BMW)
   - 提前终止不可能进入 TopK 的计算

3. 向量检索优化：
   - IVF-PQ 减少搜索空间
   - HNSW 图索引加速
```

## 8. 搜索质量评估

### 8.1 离线评估

```
指标：
- DCG (Discounted Cumulative Gain)
- NDCG (Normalized DCG)
- MAP (Mean Average Precision)
- MRR (Mean Reciprocal Rank)

人工标注：
- 相关性等级：完美/高度/一般/不相关
- 旁注法（Side-by-Side）对比
```

### 8.2 在线评估

```
指标：
- 零结果率（Zero Result Rate）
- 点击率（CTR）
- 首次点击位置（First Click Position）
- 查询 reformulation 率
- 用户停留时间

A/B 测试：
- 搜索实验通常需要更大样本量
- 关注长期指标（7 日留存）
```

## 9. 面试要点

### Q1: 搜索和推荐有什么本质区别？工程上有什么不同？

**答**：
- 搜索是主动查询，用户有明确意图，容错率低；推荐是被动接收，容错率较高
- 工程上搜索对延迟更敏感，需要复杂的 Query 理解，支持布尔查询和过滤
- 搜索的结果数量相对固定，推荐可以无限滚动

### Q2: 倒排索引如何支持布尔查询（AND/OR/NOT）？

**答**：
- AND：倒排列表求交集，使用跳表加速
- OR：倒排列表求并集
- NOT：从结果集中排除
- 复杂布尔查询可以转换为二叉树，后序遍历求值

### Q3: 如何优化长尾 Query 的搜索效果？

**答**：
1. Query 扩展：使用同义词、相关词扩展
2. 向量召回：基于语义相似度，不受关键词限制
3. 点击图：利用用户点击行为，找到 Query-Doc 关联
4. 个性化：利用用户历史行为补足冷启动

### Q4: 向量召回和倒排召回各有什么优缺点？怎么融合？

**答**：
- 倒排召回：精确匹配，可解释性强，但不理解语义
- 向量召回：语义匹配，泛化能力强，但计算量大
- 融合：RRF 算法、线性加权、或用 LTR 模型学习权重

### Q5: 搜索延迟 P99 要求 100ms，怎么做到？

**答**：
1. Query 缓存，命中率 40%，直接返回
2. 倒排索引常驻内存，SSD 存放冷数据
3. 跳表加速倒排列表求交
4. WAND/BMW 提前终止计算
5. 多路召回并行执行
6. 向量检索使用 HNSW，限制 ef 参数
