# 设计文件/图片存储系统

## 需求分析

### 功能需求

1. **文件上传**：支持大文件、断点续传
2. **文件下载**：高速下载、断点续传
3. **文件管理**：删除、复制、移动
4. **缩略图生成**：图片自动缩略
5. **CDN 加速**：全球加速访问

### 非功能需求

| 需求 | 目标 |
|------|------|
| 可用性 | 99.999%（数据不丢失）|
| 延迟 | 上传/下载 < 1s（小文件）|
| 存储规模 | PB 级 |
| 支持文件大小 | 最大 100GB |

## 容量估算

```
假设（类似腾讯云 COS）：
- 用户：1000 万
- 每人每天上传：10 个文件
- 平均文件大小：500KB（图片）
- 多媒体文件：10MB

计算：
- 日上传量：1亿 个文件
- 日存储增长：
  - 图片：1亿 × 500KB = 50 TB
  - 多媒体：假设 1% 是多媒体，100万 × 10MB = 10 TB
  - 总计：60 TB/天
- 年存储：60 TB × 365 ≈ 22 PB

QPS：
- 上传：1亿 / 86400 ≈ 1157 QPS
- 下载（假设 10 倍）：11570 QPS
- 峰值 × 3：上传 3500，下载 35000
```

## 核心设计

### 1. 文件元数据

```sql
CREATE TABLE file_metadata (
    file_id VARCHAR(64) PRIMARY KEY COMMENT '文件唯一ID（MD5）',
    bucket_id VARCHAR(64) NOT NULL COMMENT '存储桶ID',
    file_name VARCHAR(255) NOT NULL COMMENT '文件名',
    file_size BIGINT NOT NULL COMMENT '文件大小',
    content_type VARCHAR(128) COMMENT 'MIME类型',
    storage_path VARCHAR(512) NOT NULL COMMENT '存储路径',
    storage_class ENUM('STANDARD', 'IA', 'ARCHIVE') DEFAULT 'STANDARD',
    etag VARCHAR(64) COMMENT '文件MD5',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_bucket_file (bucket_id, file_name)
) ENGINE=InnoDB;

CREATE TABLE multipart_upload (
    upload_id VARCHAR(64) PRIMARY KEY,
    file_id VARCHAR(64) NOT NULL,
    bucket_id VARCHAR(64) NOT NULL,
    total_parts INT NOT NULL,
    completed_parts INT DEFAULT 0,
    status ENUM('INITIATED', 'COMPLETED', 'ABORTED') DEFAULT 'INITIATED',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    expire_at TIMESTAMP NOT NULL
);
```

### 2. 文件 ID 设计

```
方案 1：内容哈希（MD5/SHA256）
- 相同内容只存一份（去重）
- 天然防重复上传
- 但无法区分不同文件名的相同内容

方案 2：UUID
- 全局唯一
- 无去重能力

方案 3：混合（推荐）
- 文件 ID = UUID（对外）
- 存储路径 = hash(content)（对内，用于去重）
```

### 3. 大文件分片上传

```
流程：
1. 客户端请求创建分片上传任务
2. 服务端返回 uploadId
3. 客户端将文件切分为多个 part（如 5MB/片）
4. 并发上传所有分片
5. 所有分片上传完成后，请求合并
6. 服务端校验完整性，合并文件

优点：
- 断点续传：失败的分片重新上传
- 并发加速：多线程同时上传
- 秒传：如果文件已存在，直接返回成功
```

### 4. 去重机制

```
基于内容的去重：

1. 计算文件 MD5
2. 查询 Metadata 表是否存在相同 MD5
3. 存在：增加引用计数，不实际存储
4. 不存在：存储文件，记录 MD5

引用计数：
- 删除时减少引用计数
- 引用计数为 0 时才真正删除文件
```

## 存储架构

### 存储类型

| 类型 | 适用场景 | 价格 | 取回时间 |
|------|----------|------|----------|
| 标准存储 | 频繁访问 | 高 | 实时 |
| 低频存储 | 月均访问 < 1 次 | 中 | 实时 |
| 归档存储 | 很少访问 | 低 | 分钟级 |
| 深度归档 | 基本不访问 | 极低 | 小时级 |

### 存储路径设计

```
避免单目录文件过多：

路径格式：/bucket/{hash_prefix}/{file_hash}

示例：
文件 MD5: 3b5d8c9a2f1e4d6b...
存储路径: /mybucket/3b/5d/3b5d8c9a2f1e4d6b...

好处：
- 目录层级均衡
- 避免单目录文件过多影响性能
- 便于分布式存储
```

### 数据冗余

```
副本策略：
- 3 副本：标准存储
- 2 副本 + 纠删码：降低存储成本

纠删码 (Erasure Coding)：
- 将数据切分为 m 块，生成 k 块校验
- 存储 m + k 块，任意 m 块可恢复数据
- 存储效率：m/(m+k)，如 10/14 = 71%
- 相比 3 副本的 33%，大幅降低存储成本
```

## 图片处理

### 缩略图生成

```
策略：
1. 上传时预生成常用尺寸
   - 原图、大图、中图、小图、缩略图
   - 优点：访问快
   - 缺点：存储成本高

2. 访问时实时生成
   - 首次访问生成，后续缓存
   - 优点：节省存储
   - 缺点：首次访问慢

3. 混合策略（推荐）
   - 上传时生成常用尺寸
   - 特殊尺寸访问时实时生成并缓存
```

### 图片处理参数

```
URL 参数方式：
/image.jpg?width=300&height=200&mode=crop

处理模式：
- resize: 等比缩放
- crop: 居中裁剪
- fill: 填充
- watermark: 添加水印
```

## 安全和权限

### 访问控制

```
1. 私有 Bucket：
   - 需要签名 URL 才能访问
   - URL 包含过期时间
   - 服务端验证签名

2. 公有 Bucket：
   - 任何人可访问
   - 适合静态网站托管

3. Bucket Policy：
   - 精细控制访问权限
   - 支持 IP 白名单、Referer 限制
```

### 防盗链

```
Referer 检查：
- 检查 HTTP Referer 头
- 只允许特定域名引用
- 空 Referer 可配置允许/拒绝

签名 URL：
- URL 包含签名和过期时间
- 过期后自动失效
```

## 面试要点

### Q1: 如何实现大文件的断点续传？

**答**：
1. 将大文件切分为固定大小的分片（如 5MB）
2. 每个分片计算 MD5
3. 上传前查询服务端哪些分片已存在
4. 只上传缺失的分片
5. 所有分片上传完成后请求合并
6. 服务端校验分片完整性后合并

### Q2: 如何实现秒传功能？

**答**：
1. 客户端计算文件 MD5
2. 上传前将 MD5 发送给服务端
3. 服务端查询是否已有相同 MD5 的文件
4. 存在则直接返回成功，增加引用计数
5. 不存在则正常上传

### Q3: 如何降低存储成本？

**答**：
1. 数据去重：相同文件只存一份
2. 冷热分离：低频数据用廉价存储
3. 纠删码：替代多副本，提高存储效率
4. 生命周期管理：自动删除过期数据
5. 压缩：对可压缩数据启用压缩
