# 设计即时通讯系统 (IM)

## 需求分析

### 功能需求

1. **一对一聊天**：用户之间发送消息
2. **群聊**：多人群组聊天
3. **消息状态**：已发送、已送达、已读
4. **历史消息**：拉取历史聊天记录
5. **在线状态**：显示用户在线/离线
6. **多媒体消息**：图片、语音、视频

### 非功能需求

| 需求 | 目标 |
|------|------|
| 可用性 | 99.999% |
| 延迟 | 消息送达 < 1s |
| 有序性 | 消息按发送顺序到达 |
| 可靠性 | 消息不丢失、不重复 |

## 容量估算

```
假设（微信量级）：
- DAU: 10 亿
- 每人每天发 50 条消息
- 消息大小: 平均 100 bytes（文本）
- 峰值: 平均值的 2 倍

计算：
- 日消息量: 10亿 × 50 = 500 亿条
- 消息 QPS: 500亿 / 86400 ≈ 580,000
- 峰值 QPS: 120 万
- 日存储: 500亿 × 100B = 500 GB（仅文本）
- 年存储: 500 GB × 365 ≈ 182 TB
```

## 核心流程

### 消息发送流程

```
1. 用户 A 发送消息
2. Gateway 接收，校验用户身份
3. 生成全局唯一 Message ID
4. 写入消息队列 (Kafka)
5. 返回发送成功给用户 A
6. 消费者处理：
   a. 存储到 Message DB
   b. 查询用户 B 的在线状态
   c. 如果在线，通过 Gateway 推送
   d. 如果离线，发送 Push 通知
   e. 更新会话列表
```

### 消息 ID 设计

```
使用雪花算法变体：
- 保证全局唯一
- 保证趋势递增（便于排序）
- 包含时间信息

格式：
| 1位 | 41位时间戳 | 10位WorkerId | 12位序列号 |

优点：
- 本地生成，无网络依赖
- 天然按时间排序
```

### 消息有序性保证

```
问题：网络延迟可能导致消息乱序

解决方案：
1. 客户端序号：每条消息带自增序号
2. 服务端校验：发现序号不连续时，请求补发
3. 客户端排序：收到消息后按序号排序显示

或者使用服务端序号：
- 每个会话维护一个自增序号
- 消息入库时分配序号
- 客户端按服务端序号排序
```

## 在线状态管理

### 方案 1：集中式

```
使用 Redis 存储在线状态：
- user:{userId}:online → true/false
- user:{userId}:gateway → gateway_id
- user:{userId}:last_active → timestamp

查询在线状态：
GET user:{friendId}:online

心跳机制：
- 客户端每 30 秒发送心跳
- 服务端 60 秒未收到心跳，标记离线
```

### 方案 2：分布式（推荐）

```
Gateway 本地维护连接状态：
- Gateway 1: 维护用户 A, B, C 的连接
- Gateway 2: 维护用户 D, E, F 的连接

状态广播：
- 用户上线：广播给所有 Gateway
- 用户下线：广播给所有 Gateway
- 使用 Redis Pub/Sub 或 MQ

查询在线状态：
- 查本地缓存
- 未命中时查 Redis
```

## 消息存储设计

### 存储策略

```
最近消息（热数据）：
- MySQL：最近 7 天消息
- 分片：按 user_id 或会话 ID 分片
- 索引：快速查询会话最新消息

历史消息（冷数据）：
- HBase：7 天前的消息
- RowKey: hash(user_id) + timestamp + message_id
- 支持按时间范围查询

多媒体：
- 对象存储（OSS/S3）
- 数据库只存 URL
```

### 会话列表设计

```
每个用户维护一个会话列表：

MySQL 表：
CREATE TABLE conversations (
    user_id BIGINT NOT NULL,
    conversation_id VARCHAR(64) NOT NULL,
    type ENUM('single', 'group'),
    opposite_id BIGINT,          -- 对方用户ID或群ID
    last_message_id BIGINT,
    last_message_content TEXT,
    last_message_time TIMESTAMP,
    unread_count INT DEFAULT 0,
    PRIMARY KEY (user_id, conversation_id),
    INDEX idx_user_time (user_id, last_message_time)
);

Redis 缓存：
- 用户的会话列表（最近 50 个）
- TTL: 7 天
```

## 群聊设计

### 消息分发

```
群规模：
- 小群（< 100人）：直接推送到所有成员
- 大群（100-2000人）：写入扩散，成员拉取
- 超大群（> 2000人）：类似大 V 方案，只存一份

写入扩散 vs 读取扩散：

小群（写入扩散）：
- 发消息时写入每个成员的收件箱
- 读时直接读取
- 类似 Twitter 的推模型

大群（读取扩散）：
- 发消息时只存一份
- 读时查询群消息
- 类似 Twitter 的拉模型
```

### 群消息已读

```
方案 1：已读回执
- 每个人已读都发送回执
- 显示 "已读 10/50"
- 适合小群

方案 2：阈值显示
- 不显示具体已读人数
- 只显示 "已读"（超过一定人数）
- 适合大群
```

## 离线推送

### 推送策略

```
用户离线时：
1. 消息存入数据库
2. 发送 Push 通知（APNs/FCM/小米/华为）
3. 用户点击通知打开 App
4. App 拉取未读消息
5. 建立 WebSocket 连接

Push 内容：
- 简洁："张三发来一条消息"
- 或者包含消息内容（需考虑隐私）
```

### 推送限流

```
防止骚扰：
- 单个用户每分钟最多推送 N 条
- 免打扰时段不推送
- 用户可设置免打扰
```

## WebSocket 连接管理

### 连接建立

```
1. 用户登录获取 Token
2. 携带 Token 建立 WebSocket 连接
3. Gateway 验证 Token
4. 将 user_id → gateway_id 映射写入 Redis
5. 开始心跳检测
```

### 连接断开处理

```
异常断开：
1. 心跳超时检测到断开
2. 更新用户在线状态
3. 广播离线状态给好友
4. 保留连接上下文（支持快速重连）

优雅关闭：
1. 客户端发送关闭帧
2. 服务端清理资源
3. 更新状态
```

### 水平扩展

```
Gateway 集群：
- 用户 A 连接 Gateway 1
- 用户 B 连接 Gateway 2

消息路由：
1. 用户 A 发消息给用户 B
2. Gateway 1 查询 Redis：user:B → Gateway 2
3. 通过内部 RPC 转发给 Gateway 2
4. Gateway 2 推送给用户 B
```

## 消息可靠性

### 不丢失

```
发送方：
- 客户端本地存储（SQLite）
- 发送失败自动重试
- 收到服务端 ACK 后删除本地

服务端：
- 消息写入 MQ（Kafka 持久化）
- 消费者确认后才删除
- 数据库多副本

接收方：
- 收到消息后发送 ACK
- 客户端本地存储
- 展示给用户后确认
```

### 不重复

```
Message ID 去重：
- 服务端生成全局唯一 Message ID
- 客户端本地缓存最近 N 个 Message ID
- 重复消息丢弃

或者使用去重表：
- 数据库唯一索引 (user_id, client_msg_id)
- client_msg_id 由客户端生成（UUID）
```

### 消息顺序

```
单聊：
- 同一个会话的消息按时间顺序
- 使用时间戳或自增序号排序

群聊：
- 服务端分配全局递增序号
- 客户端按序号排序展示
```

## 面试要点

### Q1: 如何保证消息不丢失？

**答**：
1. 发送方本地存储 + 重试机制
2. 服务端消息队列持久化 + 多副本
3. 接收方 ACK 确认机制
4. 数据库主从复制保证数据安全

### Q2: 如何处理消息的乱序问题？

**答**：
1. 服务端为每条消息分配递增序号
2. 客户端收到消息后按序号排序
3. 发现序号不连续时，请求补发缺失消息
4. 使用时间戳作为辅助排序依据

### Q3: WebSocket 如何支持百万级并发？

**答**：
1. Gateway 集群水平扩展
2. 每个 Gateway 节点支持 10万+ 连接（使用 Netty）
3. 连接状态集中存储（Redis）
4. 消息路由通过内部 RPC 转发
5. 使用发布订阅模式广播状态变更
