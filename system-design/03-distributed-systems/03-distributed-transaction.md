# 分布式事务

## 什么是分布式事务

分布式事务是指事务的参与者、支持事务的服务器、资源服务器以及事务管理器分别位于分布式系统的不同节点上。

```
分布式事务场景：
- 跨库事务：订单库 + 库存库
- 跨服务事务：订单服务 + 支付服务 + 物流服务
- 混合事务：数据库 + 消息队列 + 缓存
```

## 理论基石

### ACID vs BASE

```
传统 ACID：
- Atomicity（原子性）
- Consistency（一致性）
- Isolation（隔离性）
- Durability（持久性）

分布式 BASE：
- Basically Available（基本可用）
- Soft state（软状态）
- Eventually consistent（最终一致）
```

### CAP 定理回顾

```
分布式系统中，Consistency、Availability、Partition Tolerance 三者不可兼得。

分布式事务通常选择：
- AP + 最终一致性（大多数互联网场景）
- CP + 强一致性（金融核心场景）
```

## 分布式事务方案

### 1. 两阶段提交 (2PC)

#### 角色

```
协调者（Coordinator）：事务管理器
参与者（Participants）：各数据库/服务
```

#### 流程

```
阶段一（Prepare）：
1. 协调者询问所有参与者是否可以提交
2. 参与者执行本地事务，锁定资源
3. 参与者返回 Yes/No

阶段二（Commit/Rollback）：
如果所有参与者返回 Yes：
   4. 协调者发送 Commit 指令
   5. 参与者提交本地事务，释放锁

如果有参与者返回 No：
   4. 协调者发送 Rollback 指令
   5. 参与者回滚本地事务，释放锁
```

#### 优缺点

```
优点：
- 强一致性
- 实现简单，理解容易

缺点：
- 同步阻塞：参与者需要锁定资源等待协调者指令
- 单点故障：协调者宕机会导致整个事务挂起
- 数据不一致：协调者宕机后重启，可能部分提交部分未提交
- 性能差：两次网络往返
```

### 2. 三阶段提交 (3PC)

#### 改进点

```
增加预提交阶段，减少阻塞时间：

阶段一（CanCommit）：
- 协调者询问参与者是否可以执行（不锁定资源）

阶段二（PreCommit）：
- 参与者预执行，锁定资源

阶段三（DoCommit）：
- 真正提交或回滚
```

#### 优缺点

```
优点：
- 引入超时机制，减少阻塞
- 协调者宕机后可以通过超时继续推进

缺点：
- 实现复杂
- 极端情况下仍可能不一致
- 网络分区时难以处理
```

### 3. TCC (Try-Confirm-Cancel)

#### 原理

将每个操作拆分为三个阶段：

```
Try：预留资源，执行业务检查
Confirm：真正执行业务（Try 成功）
Cancel：释放预留资源（Try 失败）
```

#### 示例：扣减库存

```java
// Try：检查并预留库存
boolean tryDeduct(Long productId, Integer count) {
    // 检查库存是否充足
    // 冻结待扣减的库存
    // 插入冻结记录
}

// Confirm：确认扣减
boolean confirmDeduct(Long productId, Integer count) {
    // 实际扣减库存
    // 删除冻结记录
}

// Cancel：取消扣减
boolean cancelDeduct(Long productId, Integer count) {
    // 释放冻结的库存
    // 删除冻结记录
}
```

#### 优缺点

```
优点：
- 最终一致性，性能较好
- 无锁设计，并发度高
- 适合长事务

缺点：
- 业务侵入大，每个操作要拆成三个
- 实现复杂
- Confirm/Cancel 需要保证幂等
```

#### TCC 框架

```
阿里 Seata：
- 开源分布式事务框架
- 支持 AT、TCC、SAGA、XA 模式
- 零侵入的 AT 模式（基于 SQL 解析）
```

### 4. Saga 模式

#### 原理

```
将长事务拆分为多个本地事务：
- 每个本地事务有对应的补偿操作
- 如果某个本地事务失败，执行前面所有事务的补偿

两种实现方式：
1. 编排式（Choreography）：事件驱动，各服务监听事件
2. 编排式（Orchestration）：中央协调器统一调度
```

#### 示例：电商下单

```
本地事务序列：
T1: 创建订单
T2: 扣减库存
T3: 扣减余额
T4: 创建物流单

如果 T3 失败：
C3: 恢复余额
C2: 恢复库存
C1: 取消订单
```

#### 优缺点

```
优点：
- 适合长事务（业务流程长）
- 一阶段提交本地事务，性能好
- 天然适合微服务架构

缺点：
- 隔离性差（中间状态可见）
- 补偿逻辑复杂
- 可能出现补偿失败（需要人工介入）
```

### 5. 本地消息表

#### 原理

```
将分布式事务转为本地事务 + 消息投递：

1. 业务表和消息表在同一个数据库
2. 业务操作和消息插入在同一个本地事务
3. 后台任务扫描消息表，投递到 MQ
4. 消费者消费并处理业务
5. 处理成功后删除或标记消息
```

#### 表设计

```sql
-- 消息表
CREATE TABLE local_message (
    id BIGINT PRIMARY KEY,
    biz_type VARCHAR(32),       -- 业务类型
    biz_id VARCHAR(64),         -- 业务ID
    payload TEXT,               -- 消息内容
    status TINYINT,             -- 0:待发送 1:发送中 2:成功 3:失败
    retry_count INT,            -- 重试次数
    create_time TIMESTAMP,
    update_time TIMESTAMP
);
```

#### 优缺点

```
优点：
- 实现相对简单
- 最终一致性
- 不依赖外部框架

缺点：
- 需要维护消息表
- 需要定时任务扫描
- 消费方需要幂等
```

### 6. 最大努力通知

#### 原理

```
适用于对一致性要求不高的场景：

1. 业务系统执行本地事务
2. 发送通知到 MQ
3. 通知服务消费并通知第三方
4. 如果通知失败，按策略重试（固定间隔、指数退避）
5. 达到最大重试次数后，停止或人工介入

典型场景：支付回调通知
```

## 方案对比与选型

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| 2PC | 强一致 | 低 | 中 | 传统单体拆分时 |
| 3PC | 强一致 | 低 | 高 | 较少使用 |
| TCC | 最终一致 | 高 | 高 | 高并发、短事务 |
| Saga | 最终一致 | 高 | 中 | 长事务、业务流程 |
| 本地消息表 | 最终一致 | 高 | 中 | 异步场景 |
| 最大努力通知 | 弱一致 | 高 | 低 | 对账类场景 |

## 互联网实践

```
阿里：
- Seata（AT 模式为主，配合 TCC）
- 本地消息表（内部早期方案）

美团：
- 自研 Saga 框架
- TCC 用于核心交易

通用原则：
- 能不用分布式事务就不用
- 优先使用异步 + 最终一致
- 核心链路用强一致，非核心用最终一致
```

## 面试常见问题

### Q1: 2PC 和 TCC 有什么区别？

**答**：
- 2PC 是资源层的协议，通过锁定资源保证原子性，性能差但实现简单
- TCC 是业务层的补偿机制，通过预留+确认/取消实现，性能好但业务侵入大

### Q2: 分布式事务如何保证幂等？

**答**：
- 唯一键去重：数据库唯一索引
- 状态机幂等：根据状态判断是否已处理
- Token 机制：先申请 Token，处理时携带，处理成功删除 Token
- 乐观锁：version 字段控制

### Q3: 实际项目中如何选择分布式事务方案？

**答**：
- 首先评估是否真的需要分布式事务，能否通过业务设计避免
- 金融核心场景（转账）：2PC 或 TCC 保证强一致
- 电商下单：Saga 或 TCC，优先可用性
- 异步通知场景：本地消息表或最大努力通知
