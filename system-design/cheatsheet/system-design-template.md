# 系统设计面试模板

## 4S 框架

### 1. Scope（明确范围）- 5分钟

#### 功能性需求
- 核心功能是什么？
- 必须支持哪些功能？
- 可以暂时不支持哪些功能？

#### 非功能性需求
```
QPS：每秒查询率
- 读 QPS：___
- 写 QPS：___
- 峰值倍数：___

存储：
- 数据总量：___
- 日增长：___
- 数据类型：文本/图片/视频

延迟要求：
- P99 读延迟：___
- P99 写延迟：___

可用性：___ 个 9
```

#### 容量估算公式

```
QPS = DAU × 人均日操作次数 / 86400
存储 = 数据量 × 单条大小 × 保留时间
带宽 = QPS × 平均数据大小
```

### 2. Sketch（高层设计）- 10分钟

#### 画架构图
```
- 接入层：LB、CDN、API Gateway
- 服务层：核心服务、辅助服务
- 数据层：数据库、缓存、消息队列
- 外部依赖：第三方服务
```

#### 定义 API
```
RESTful API 设计：
POST   /api/v1/resource      - 创建
GET    /api/v1/resource/{id} - 查询
PUT    /api/v1/resource/{id} - 更新
DELETE /api/v1/resource/{id} - 删除
```

#### 数据模型
```
- 核心实体
- 实体关系
- 主键、索引设计
```

### 3. Solidify（详细设计）- 15分钟

#### 核心组件深入
```
- 数据库选型：SQL vs NoSQL
- 缓存策略：Cache Aside vs Write Through
- 分片策略：哈希 vs 范围
- 一致性保障：强一致 vs 最终一致
```

#### 处理边界情况
```
- 热点数据
- 数据倾斜
- 故障场景
- 并发冲突
```

### 4. Scale（扩展性分析）- 10分钟

#### 当前瓶颈
```
- 数据库：读压力大？写压力大？
- 缓存：命中率低？容量不足？
- 服务：CPU 瓶颈？IO 瓶颈？
```

#### 扩展方案
```
- 水平扩展：加机器
- 垂直扩展：升配
- 读写分离
- 分库分表
- CDN 加速
```

#### 故障处理
```
- 单点故障消除
- 降级策略
- 熔断限流
- 监控告警
```

## 面试技巧

### Do's

1. **先问后答**：澄清需求再开始设计
2. **自顶向下**：先给整体架构，再深入细节
3. **量化分析**：给出具体的数字估算
4. **讨论权衡**：说明不同方案的优缺点
5. **保持互动**：与面试官持续沟通

### Don'ts

1. **不要沉默**：边想边说，展示思考过程
2. **不要钻牛角尖**：先给可行方案，再优化
3. **不要过度设计**：先满足需求，再考虑扩展
4. **不要忽视非功能需求**：性能、可用性很重要
5. **不要只谈技术**：考虑业务场景

## 常见问题及回答思路

### Q: 如何设计一个支持 10 亿用户的系统？

**思路**：
1. 分层架构：接入层、服务层、数据层
2. 水平扩展：无状态服务 + 分片存储
3. 缓存：多级缓存减少数据库压力
4. CDN：静态内容分发
5. 异地多活：全球部署

### Q: 如何保证系统高可用？

**思路**：
1. 消除单点：多副本部署
2. 故障隔离：服务熔断、限流
3. 自动恢复：自动故障转移
4. 监控告警：及时发现问题
5. 降级策略：故障时保证核心功能

### Q: 如何处理热点数据？

**思路**：
1. 缓存：多级缓存，本地 + 分布式
2. 复制：热点数据多副本
3. 打散：分片、一致性哈希
4. 限流：保护热点数据源
5. 预热：提前加载热点数据
